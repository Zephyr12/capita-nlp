Data Mining Pipelines

The purpose of the data mining pipeline is to grant us access to social media data, in a useful and accessible format, that is reliably and periodically captured. It focuses on capturing a variety of social media education-related data that include information about specific educational institutions or people in the educational domain, from relevant social media platforms.

Our information is updated daily. However the process can be done more often but it would be counter productive as it comes with unwanted risks: receiving more data than the system can process, which would result in a system failure.

The three initial social media sources were Twitter, Facebook and MumsNet. Lack of content on MumsNet forum, forced us to switch to another source and hence, we chose scraping The Student Room - which required the same scraping technology. Morever, every source required us to consider a specific API to work with.


Facebook SDK for Python

WHY FACEBOOK API?

In order to create a Facebook pipeline, we had a few options: either using the Facebook provided SDK for Python[1], or use other libraries that would ease the work with the API for the developers. Although using other libraries would have made our initial developing process more easy (achieving a working Facebook pipeline), it would have come with risks that we could not take such as unwanted and unknown specific bugs inside that library, as they are not meant to be used for scraping. Considering our system complexity and the fact that Data Pipeline is our initial component, we decided on using the Facebook provided API, despite its poor documentation and deprecated functionalities in time[2].


RAISED ISSUES

As we have specified the deprecated functionalities, we came to realise that Facebook did provide support for Public Feed API (which was our main request for scraping the desired information), but they restrictedthe access to the Feed API back in 2015 - which left us wondering what solution is most appropriate in order to get the desired information from Facebook.

SOLUTION

Even though we could not make use of the public feed of Facebook posts, we could still make search query requests to the Facebook API. As a result, we can have access to search events that are related to any UK Educational Institution. Thus, this pipeline is set up to yield event descriptions (together with other relevant information as concerned university, location, dates).

Scraping The Student Room with SCRAPY

WHY SCRAPY?

In order to store all the posts relating the Universities in UK that are found on The Student Room forum, we had two options: Scrapy[3] or Beautiful Soup.

While BeautifulSoup is just a library for parsing html(and xml) style documents, Scrapy is a full framework for web crawling which provides the tools to manage every stage of web crawl - tools that come with certain advantages. Some of these tools are:

  1. Request manager: provides the speed of cuncurrency without needing to invest time in developing the concurrent architecture, as it is all done behind the scenes.
  2. Selectors: which are used to parse the HTML documents in order to find the bits that are needed to be parsed. (BeautifulSoup also does this).
  3. Pipelines: once the data is retrieved, it can be passed through various pipelines, which serves the purpose of our application.

As a result, we decided that Scrapy will be not only easier to implement (which also brings the advantage of being easier to debug and maintain), but actually faster because of the way it handles cocunrrency.


RAISED ISSUES

Because of the way Scrapy works, even though it produced the desired output on it's own (yielding forum posts), it did not behave as expected when we tried to merge the pipelines. Scrapy runs more threads in order to be faster and more efficient, but in order to be able to send signals between these threads, it needs to run on the main one in order - which it didn't at the stage of merging the pipes.

SOLUTION

Looked up the Source Code for Scrapy to understand how it starts its crawler and then modified the thread invocation.

Data Transformation and Feature Extraction

These technologies are related to the meaning of the data that is fetched by the data mining pipeline. In other words, these are responsible for producing the insight about the data that we are collecting.

We used gensim for topic modelling and nltk for Named-Entity Recogniser(NER) and Sentiment Analysis.

GENSIM

WHY GENSIM?

Gensim[5] is a free Python library designed for topic modelling. It presents a lot of advantages and seemed to be the most accessible technology for our purpose at this level. Some of the useful advantages for our system are:

  • Scalability: Gensim can process large, web-scale corpora, using incremental online training algorithms. There is no need for the whole input corpus to reside fully in RAM at any one time.
  • It is Platform Independent: Being pure Python, gensim runs on Linux, Windows and OS X, as well as any other platform that supports Python and NumPy.
  • Provides efficient implementations: The core algorithms in gensim use highly optimized math routines. Gensim also contains a distributed version of several algorithms, intended to speed up processing and retrieval on machine clusters.


RAISED ISSUES

Because of the complexity of this advanced process, gensim lda models needed a lot of time and data for training - resources that were not available for the scale of our system.

SOLUTION

We used Google's pre-trained word2vec corpus in order to achieve the necessary output from our system.

NLTK

WHY NLTK?

Our options for language processing were spaCy and nltk[6]. The difference between these two technologies is that spaCy represents an Industrial-Strenght Natural Language Processing while NLTK is designed to be a good starting point in this area (students, researchers).

Moreover, spaCy is faster at Word Tokenization and Part-Of-Speech Tagging, while NLTK is faster at Sentence Tokenization - the last one being our system scope. Thus, we considered that in our case it is would be the best decision to chose NLTK over spaCy as it provides more support and is faster in the area that are mostly concerned about.
