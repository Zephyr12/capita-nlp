API Interaction



Our system is designed to be an API, which makes it to be designed by humans for humans. Hence, this is how it was developed to be easy accessible by future developer.

Short Overview

As every system is designed to be further used by other humans (either users or developers), this is our alternative of explaining the HCI component of our system: the main purpose of our project is to create an useful API, so the human-computer interaction will take place between the developer and the system API (developer-code interaction). Keeping this in mind, the system was designed to fulfil a couple of important features in order to make it easy-to-use by developers.

Some of these features represent:

  1. Explicit code structures
  2. API surface area is not bigger than necessary
  3. Reduced dependencies
  4. Error returns have a meaningful state
  5. Functionalities are well documented
  6. Has written tests





Explicit Code Structures

Every function is designed to do what it's name says it does. Function names are explicit and they represent the purpose of the method. The purpose is to avoid creating functions that also do what they are expected to do but might have some unknown side-effects - such that a future user of our API would have to dive deep in the source code to understand why his implementation does not behave as desired. Here are some examples of function names, together with their parameter names:

Small API Surface Area

Our system is composed of many component, but it shouldn't be the responsibility of any future developer to manage each component. In fact, as a future developer, he should be able to access the main features of the system without interacting with it's components too much.




Reduced Dependencies

As it is presented on our Design page, the system reduces the dependencies as much as possible, keeping the code self-contained where possible. That is because the more dependencies a system has, the more potential problems it can cause id downstream consumer code.

Where it has to use pieces of functionalities from other modules, it only extracts those pieces of functionality that are required. Thus, the system balances between code reuse and tight coupling - trying to decide when is best to rewrite a piece of code (if it is a small one for example) a and when to extract one from another module(when the task is more complex).




Meaningful Error States

If a developer encounters a problem with it's code implementation, he should be able to understand what went wrong so that he can debug his code. However, if the return state would not contain any relevant piece of information (explaining what caused the error), his only chance to fix it is to start digging inside the source code and try to understand where does the problem come from.
