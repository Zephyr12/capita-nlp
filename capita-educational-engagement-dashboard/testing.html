Testing


These are the testing strategies that we consider are necessary to check if our system behaves as expected and is resilient.





Stress Testing

The purpose of the test stressing is to assert that our system works as expected under normal environmental conditions. If the system passes a stress test (exaggerated levels of stress), then we can assume that it would not fail under normal conditions.





Unit Testing

Unit Testing represents the most pervasive kind of testing for our system - checking if small individual components do behave as expected. They are also useful for the development process as they reassure the creators of the system that their implementations are correct, but it also makes it easier to add new functionality - that is because it offers the certainty that the old behaviour of the system did not change despite the changes that were made.

To test our Data Processing Pipeline, we used Python's nose2 module in combination with the unit testing framework to ensure fast and repeatable unit tests.

We used Scrapy Contracts for The Student Forum spider to assure that the spider work as intended. A Scrapy Contract assures that the parse level returns a certain level of items (some levels should not return items, while others should return at least one item) and that it does scrape and return the correct information (the forum post and the concerned university).





Integration Testing

This stage represents the testing of multiple individual modules combined together and checked if they behave accordingly to the specifications as a group. It takes our individual modules that were unit tested and then applies tests defined in the integration test plan and then delivers as its output the integrated system, assuring that the components work as expected when merged.
