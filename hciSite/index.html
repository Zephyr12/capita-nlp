  • Capturing and evaluating social media data sources related to educational domain

  • Transforming raw data (Facebook posts, tweets) into an interactive Dashboard

  • Discover trends and patterns about schools, parental engagement and local areas

    • MoSCoW


    Must Have
    • Capture data about specific organisations and people in the education domain from Facebook and Twitter
    • Transform data into a more consumable format to make it easier to discover associations and report on
    • Understand what patterns are present in social media data when related to education
    • Provide Power BI Dashboard/Report

    Should Have
    • Data Agnostic, accepts data definitions, source definitions, extraction plugins and routes the data into an appropriate storage
    • Standardise fields for reporting
    • Use openly available data where needed
    Could Have
    • Enrich our mined social media data with Geographical information
    • Identify useful extensions to the data (e.g. Derived columns for hashtags in tweets, extract URLs from Facebook posts)
    • Transform data into a Dimensional Model

    Questionnaire Findings

  • We have chosen to deliver this questionnaire to students, parents and teachers instead of Marketers or Product Managers (these two being our final users).
  • This is because our project is aimed at providing access to various features for the final users from different perspectives (as a student, teacher or parent).
  • The results also helped us to change the initial wireframes, as we realised we need a claner way to deliver the listed below features to the final user.



    Expected features (in order of importance):


    1.) School summaries
    • Ranking/Score
    • Student Satisfaction
    • Teacher Satisfaction
    • Social Media Trends
    2.) Filtering
    • Location Based
    • Name based Search
    3.) Parental Engagement Analysis

    Prototyping

    • We created our prototype using HTML5, CSS3 and JavaScript
    • It was chosen for it's short iteration time and ability to easily script behavior and provide visual cues
    • We decided to not use a bespoke prototyping tool as we were pressed for time, due to delays from our client, and had no time to spend learning one.
    • We decided not to use Flash as it is starting to be phased out of the Web and our team works across multiple platforms so a native solution was infeasible
